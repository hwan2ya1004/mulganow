# 홍보팀 준비 현황 노트 (2026-07-11 작성)

물가나우 블로그 자동 발행에 딸린 SNS/네이버 확산 작업의 현재 상태와 남은 준비 단계를
정리한 문서입니다. **이 문서는 조사·점검용이며, 실제 게시/발송 실행은 포함하지 않습니다.**

---

## 1. 네이버 블로그 메일 발송 로직 검토 (`generate_blog_post.py`)

대상 코드: `render_naver_draft()` (L835-878), `send_naver_draft_email()` (L881-906),
`main()`의 발송 호출부 (L980-992). 워크플로우: `.github/workflows/generate-blog-post.yml`.

### 확인 결과: 문제 없음

- **한글 제목/본문 인코딩** — `MIMEText(body, _charset="utf-8")`로 본문을 생성하면
  `Content-Transfer-Encoding: base64`가 자동 적용되고, `msg["Subject"] = subject`처럼
  한글이 섞인 문자열을 헤더에 그대로 대입해도 `email.header.Header`가 인코딩 시
  us-ascii 인코딩 실패를 감지해 자동으로 UTF-8 `=?utf-8?b?...?=` 형식으로 폴백합니다.
  실제로 스크래치패드에서 재현 테스트를 돌려 `msg.as_string().encode("ascii")`가
  예외 없이 통과하는 것을 확인했습니다 (smtplib.sendmail이 내부적으로 하는 것과 동일한 검증).
  → **헤더/본문 모두 SMTP 전송 시 깨지지 않습니다.**
- **첨부 이미지 처리** — 이미지는 첨부/임베드하지 않고 순수 텍스트만 보냅니다.
  네이버 블로그에 사람이 옮겨 붙일 때 이미지는 어차피 직접 골라 올려야 하므로
  (애드픽 상품 썸네일을 그대로 쓰면 안 됨, 아래 2번 체크리스트 참고) 의도된 설계로 보입니다.
- **실패 시 처리** — `send_naver_draft_email()`은 환경변수 미설정 시 조용히 `False`를
  반환하고, `main()`은 발송 전체를 `try/except`로 감싸 실패해도 블로그 발행 자체(파일
  커밋 대상)에는 영향이 없도록 되어 있습니다. `print()`로 성공/건너뜀/실패를 모두
  로그에 남기므로 GitHub Actions 로그에서 추적 가능합니다.
- **재시도 로직 없음** — SMTP 연결이 일시적으로 실패하면(타임아웃 20초 설정돼 있음)
  재시도 없이 그 회차는 실패로 끝납니다. 주 2회(월/목) 발행 빈도를 감안하면 심각한
  문제는 아니지만, 실패가 잦아지면 다음 발행까지 원고를 못 받는 공백이 생깁니다.
  → **개선 아이디어(참고용, 미실행)**: 실패 시 1회 재시도 또는 발행된 글 자체에
  "이번 회차 메일 발송 실패" 표시를 남기는 정도는 저비용으로 추가 가능해 보입니다.
- **발신 계정 보안** — Gmail 앱 비밀번호 방식(SMTP AUTH)이라 OAuth보다 관리가
  단순하지만, 유출 시 위험도 큽니다. GitHub Actions 시크릿으로만 보관되고 있는지
  (레포 어디에도 하드코딩 안 됨) `.env`, `.env.example`은 이 스크립트와 무관하므로
  별도 확인은 하지 않았습니다.

### 결론
로직 자체는 **실전 투입 가능한 상태**로 보입니다. 다만 실제 메일이 `seohilab@naver.com`
수신함에 도착했는지(스팸함 포함)는 마스터가 아직 육안 확인을 못 했으므로, 다음 발행
회차(다음 월/목) 이후 **네이버 메일함을 직접 열어 도착 여부·형식이 깨지지 않았는지
확인하는 절차가 남아있습니다.**

## 2. 네이버 블로그 수동 게시 체크리스트

`2026-07-10-price-watch.html`, `2026-07-13-chobok-samgyetang.html` 두 글을 참고해
사람이 메일로 받은 원고를 네이버 블로그에 옮길 때 확인해야 할 항목입니다.

### 제목 (SEO)
- [ ] 원고 제목을 그대로 쓰기보다, 네이버 검색 특성(키워드 앞쪽 배치)에 맞게
  "초복 삼계탕 재료값" 같은 핵심 키워드를 제목 **앞부분**으로 옮기는 것을 고려
  (예시 글의 제목 "초복 코앞인데 삼계탕 재료값이 뚝? 닭·마늘·전복 최저가 총정리"처럼
  이미 계절 키워드가 앞에 오는 경우가 잘 뽑힌 사례)
- [ ] 제목 글자 수 25자 내외(네이버 검색결과 잘림 방지)
- [ ] "물가나우", "KAMIS", "참가격" 등 브랜드/출처 키워드를 본문 어딘가에는 포함

### 태그
- [ ] 메일 원고 맨 아래 해시태그(`#물가정보 #물가나우 #장바구니물가 #품목명`)를
  네이버 블로그 "태그 편집"란에 그대로 입력 (본문에 이모지로 남겨도 되지만 네이버는
  별도 태그 필드가 검색 노출에 더 유효)
- [ ] 절기 이벤트가 걸려있으면(예: 초복/장마/추석) 해당 절기 키워드 태그 추가
  (`generate_blog_post.py`의 `SEASONAL_EVENTS`에 정의된 이벤트명과 맞추면 일관성 유지)

### 카테고리
- [ ] 네이버 블로그 카테고리를 "생활/노하우" 또는 "쇼핑" 등 물가 정보성 카테고리로 고정
  (미분류 상태로 두면 검색 반영이 늦어짐)
- [ ] 농산물 글 / 생필품 글을 구분해 하위 카테고리를 나누는 것도 고려 가능
  (원고에 `kind`가 produce/consumer로 이미 구분되어 있어 메일 제목이나 본문에서
  어떤 유형인지 파악 가능)

### 이미지 저작권 (가장 주의 필요)
- [ ] **메일 원고에는 이미지가 첨부되지 않습니다.** 홈페이지 블로그 글(html)에는
  `shop2.daumcdn.net` 등 쇼핑몰 CDN에서 가져온 애드픽 제휴 상품 썸네일이 쓰이는데,
  이건 애드픽 제휴 프로그램 약관상 **웹사이트(mulganow.vercel.app) 게시에는
  허용**되지만, 네이버 블로그처럼 별도 플랫폼에 그대로 재업로드해도 되는지는
  애드픽 이용약관을 다시 확인 필요 (제휴사 썸네일 재배포 범위가 플랫폼별로 다를 수 있음)
- [ ] 안전한 대안: 물가나우 자체 로고/OG 이미지(`og-image.svg`)나 텍스트 카드형
  자체 제작 이미지를 대표 이미지로 쓰고, 상품 썸네일은 링크만 텍스트로 안내
- [ ] 무료 이미지(예: 직접 촬영, 혹은 라이선스가 명확한 스톡 이미지)를 대표 이미지로
  추가하는 것을 권장 — 네이버는 대표 이미지 없는 글의 검색 노출이 불리함

### 링크/제휴 고지
- [ ] 원문 하단 `disclosure`(애드픽 제휴 마케팅 고지) 문구를 네이버 글에도 반드시 포함

## 3. 페이스북 페이지 / Threads 자동 게시 준비 리서치 (순수 조사, 미실행)

### 공통 절차 개요
1. **Meta 개발자 계정 생성**: developers.facebook.com에서 일반 페이스북 계정으로
   개발자 등록 (비즈니스 인증 필요할 수 있음)
2. **앱(App) 생성**: 앱 유형은 "비즈니스(Business)" 선택 권장. Threads를 함께 쓸
   경우 앱 타입을 "Other" → Threads 유스케이스를 제품 목록에서 추가
3. **Facebook 페이지 연결**: 이미 존재하는 물가나우 공식 페이지를 앱과 연결
   (없다면 페이지부터 먼저 생성 필요)
4. **권한(permission) 요청 및 앱 검수(App Review)**
5. **토큰 발급 및 저장** (GitHub Actions 시크릿으로 관리, 기존 GMAIL_* 시크릿과 동일한 방식)
6. **정기 토큰 갱신 자동화** (cron 워크플로우 또는 발행 워크플로우에 갱신 로직 포함)

### 3-1. Facebook 페이지 자동 게시

- **필요 권한**: `pages_manage_posts`(게시글 작성), `pages_manage_metadata`,
  `pages_read_engagement`, `pages_show_list`(계정이 관리하는 페이지 목록 조회)
- **앱 검수(App Review)**: 위 권한들은 기본 제공 범위를 넘어서는 "고급 액세스"이므로
  Meta 앱 검수 제출이 필요합니다. 검수 시 실제 사용 사례를 보여주는 스크린캐스트,
  개인정보처리방침 URL, 앱의 실제 사용 목적 설명이 요구됩니다. 불필요한 권한을
  같이 요청하면 반려 사유가 되므로 꼭 필요한 4개만 요청 권장.
- **토큰 발급 흐름**:
  1) 사용자(관리자) 로그인 → 단기 사용자 액세스 토큰 발급
  2) 단기 토큰을 장기 사용자 토큰(만료 약 60일)으로 교환
  3) 장기 사용자 토큰으로 `/{user_id}/accounts` 호출 → 페이지별 **페이지 액세스 토큰** 획득
  4) 이 페이지 액세스 토큰은 "장기 사용자 토큰이 유효한 한" 만료되지 않는 것으로
     간주되지만(사실상 무기한에 가까움), 실무적으로는 사용자 토큰 자체가 60일마다
     만료되므로 **60일 주기로 전체 토큰 재발급 절차를 자동/수동으로 반복해야 함**
  5) 게시: `POST /{page_id}/feed` (message, link 등 파라미터)
- **참고**: Page Access Token 자체의 만료 정책은 상황에 따라 다르다는 보고가 많아
  (일부는 무기한, 일부는 60일) 실제 발급 후 만료 여부를 직접 테스트해보는 것이
  안전합니다 — 이 부분은 문서만으로 100% 확정하기 어려움.

### 3-2. Threads 자동 게시

- **필요 권한**: `threads_basic`(기본, 필수), `threads_content_publish`(게시 발행).
  프로필 조회/인사이트가 필요하면 `threads_manage_insights`, `threads_read_replies` 등 추가.
- **앱 검수**: Threads API도 프로덕션(자기 계정 외 사용) 전환 시 앱 검수 필요.
  단, 개발자 본인 계정(Threads Tester로 등록된 계정) 대상 테스트는 검수 없이 가능.
- **게시는 2단계 프로세스**:
  1) `POST /{threads_user_id}/threads` — 미디어 컨테이너 생성 (텍스트/이미지/링크 등)
  2) `POST /{threads_user_id}/threads_publish` — 생성된 컨테이너를 실제 게시
- **토큰 수명**:
  - 단기 토큰: 발급 후 1시간 유효
  - 단기 → 장기 토큰 교환: `GET /access_token` (`grant_type=th_exchange_token`, 서버사이드에서만 수행, 앱 시크릿 필요)
  - 장기 토큰: 발급 후 60일 유효
  - 갱신: `GET /refresh_access_token` (`grant_type=th_refresh_token`) — **토큰 발급 후
    24시간이 지나야 갱신 가능하며, 갱신하면 다시 60일 연장**됨
  - **60일 동안 한 번도 갱신하지 않으면 토큰이 완전히 만료되어 재발급(재로그인)부터
    다시 해야 함** → 물가나우처럼 주 2회 자동 발행 파이프라인에 얹으려면, 매 발행 시
    (또는 최소 50일 주기 별도 cron으로) 토큰을 미리 갱신해서 GitHub Actions 시크릿에
    다시 저장하는 자동화가 필요 (시크릿 값을 워크플로우가 스스로 갱신하려면
    `gh secret set` 같은 별도 스텝 + PAT 권한이 필요함 — 이 부분이 구현 난이도가 있음)

### 3-3. 두 플랫폼 공통으로 남은 일 (실행 안 함, 목록만)

- [ ] Meta 개발자 계정 생성 및 비즈니스 인증
- [ ] 물가나우 공식 Facebook 페이지 존재 여부 확인 (없으면 신규 생성)
- [ ] Meta 앱 생성 (Business 타입) + Threads 유스케이스 추가
- [ ] 개인정보처리방침 페이지 URL 준비 (앱 검수 필수 항목 — mulganow.vercel.app에
  기존 개인정보처리방침이 있는지 확인 필요, 없으면 신규 작성 필요)
- [ ] 필요 권한만 선별해 앱 검수 제출 (`pages_manage_posts` 등 4종 / `threads_basic`,
  `threads_content_publish`)
- [ ] 앱 검수 통과 후 페이지 액세스 토큰 + Threads 장기 토큰 최초 발급
- [ ] 두 토큰을 GitHub Actions 시크릿(`FB_PAGE_TOKEN`, `THREADS_ACCESS_TOKEN` 등 가칭)으로 등록
- [ ] 토큰 자동 갱신 워크플로우 설계 (특히 Threads는 60일 만료 — 최소 50일 주기 갱신 필요)
- [ ] `generate_blog_post.py`에 발행 후 SNS 게시 호출 추가 여부 결정 (네이버 메일
  발송과 동일한 위치, `main()` 마지막 블록에 이어 붙이는 구조가 자연스러워 보임)

## 4. 요약 — 무엇이 준비됐고 무엇이 남았나

| 채널 | 상태 | 비고 |
|---|---|---|
| 웹사이트 블로그 자동 발행 | ✅ 완료 | 월/목 자동 실행 중 |
| 네이버 블로그 원고 메일 발송 | ✅ 구현·배포 완료, 코드 검토 결과 이상 없음 | **실제 메일함 도착 확인만 남음** (다음 발행 회차 때 확인 필요) |
| 네이버 블로그 수동 게시 | ⬜ 사람이 직접 수행 | 본 문서 2번 체크리스트 참고 |
| Facebook 페이지 자동 게시 | ⬜ 미착수 | Meta 앱 생성부터 시작 필요 (본 문서 3-1) |
| Threads 자동 게시 | ⬜ 미착수 | Meta 앱 생성부터 시작 필요 (본 문서 3-2) |

이 문서는 실제 앱 생성, 토큰 발급, 게시/발송을 수행하지 않은 **조사 전용** 자료입니다.

## 5. 추가 메모 (2026-07-19 작성) — 수동 발행 시 놓치기 쉬운 두 가지

오늘(2026-07-19) 작성팀이 `generate_blog_post.py`를 거치지 않고
`frontend/blog/2026-07-19-summer-dryness-care.html`을 수동으로 작성한 사례를 홍보팀이
처리하면서 확인한 점입니다.

- **네이버 원고 메일은 스크립트 실행 시에만 나갑니다.** `render_naver_draft()`/
  `send_naver_draft_email()`은 모두 `main()` 안에서만 호출되므로, 글을 수동으로 추가하면
  이 경로를 완전히 건너뜁니다. 오늘처럼 홍보팀이 새 글 내용을 보고
  `naver_drafts/latest.md`를 수동으로 다시 작성해 채워 넣어야 그다음 git push 시
  메일이 나갑니다. **자동 발행이 아닌 회차마다 이 수동 단계를 빠뜨리지 않도록 주의가 필요합니다.**
- **`blog/index.html`(카드 목록)·`frontend/sitemap.xml`도 스크립트가 자동으로 갱신해주던
  부분입니다.** 오늘 작성팀이 처음엔 두 파일에 새 글을 반영했으나, 작업 도중 다른 세션에서
  `git checkout` 등으로 되돌려진 것을 확인하고 그 상태를 존중해 다시 반영하지 않았습니다.
  그 결과 이 노트를 쓰는 시점 기준 **`2026-07-19-summer-dryness-care.html`은 블로그
  목록(`/blog`)과 사이트맵 어디에도 링크되어 있지 않은 상태**입니다(글 파일 자체는 존재).
  검색 노출·내부 도달성에 영향이 있으므로, 마스터가 의도한 되돌림이 맞는지 확인 후
  필요하면 두 파일에 새 글 항목을 다시 추가해야 합니다.
