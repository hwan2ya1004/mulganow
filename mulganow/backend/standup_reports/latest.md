## 0. 긴급 이슈 — 사용자가 직접 겪은 KAMIS 406 오류, 원인 파악 및 수정 완료

세션 도중 사용자가 라이브 사이트에서 직접 본 오류 메시지를 전달함:
`⚠️ 데이터를 불러오지 못했습니다: KAMIS API 호출 실패: 406 Client Error: Not Acceptable ...`

**원인**: `vercel.json`에 `regions` 설정이 없어 Vercel 서버리스 함수가 기본 리전인
**미국(iad1, 워싱턴 D.C.)**에서 실행되고 있었음(`vercel inspect`로 직접 확인). KAMIS(한국
정부 공공데이터) 서버가 해외/클라우드 데이터센터 IP 대역을 WAF 차원에서 거부하는 것으로
추정 — 동일한 요청을 국내 IP(마스터·개발팀 각자 로컬, `AS4766 Korea Telecom`)에서 보내면
**항상 200 정상**이었고, User-Agent 헤더 유무는 4가지 조합 모두 결과에 영향 없음을
개발팀이 재현 테스트로 직접 반증함.

**조치**: `vercel.json`에 `"regions": ["icn1"]`(서울)을 명시해 함수 실행 리전을 고정.
`kamis_client.py`는 BASE_URL을 https 직접 지정(불필요한 리다이렉트 제거) + 브라우저
User-Agent 헤더 추가(보조 조치, 필수 원인은 아니었음).

**마스터 직접 배포·검증**:
- 커밋 `3d75ec8` → 원격 저장소의 무관한 자동 커밋 2건(생필품 스냅샷 갱신)과 병합 → 푸시 `359d4ba`
- 배포 완료 후 재확인: `curl "https://mulganow.vercel.app/api/today-prices?cls=01"` →
  **HTTP 200, 실제 138개 품목 데이터 정상 반환** (예: 쌀/20kg 61,555원). 오류 완전 해소 확인.

## 1. 부수적으로 발견·수정한 버그 — 자동 발행 파이프라인 조용한 전면 스킵 위험

기획팀이 GA4 분석 중 `generate_blog_post.py`의 `main()`(L921-926)을 검토하다 발견:
KAMIS 호출이 실패하면 생필품(참가격) 폴백을 시도하기도 전에 함수가 즉시 `return`하는
구조였음. 즉 **KAMIS가 죽어 있는 동안은 참가격 데이터가 멀쩡해도 자동 발행이 통째로
스킵될 위험**이 있었음(다음 자동 발행일은 7/21·7/24 화/금 — 정정: cron은 UTC 월/목이지만
KST 환산하면 실제로는 화/금요일 발행. 기존 문서의 "월/목" 표기는 오기였음, 기획팀이 발견).
`except` 블록에서 `return` 대신 `enriched = []`로 바꿔 생필품 폴백 로직이 계속 실행되도록
수정, `python -m py_compile`로 검증 후 위 커밋에 포함해 배포함.
→ **확인 못한 것**: 이 수정이 실제 자동 발행에서 의도대로 동작하는지는 다음 정기 발행일
(7/21 화)에 실제로 글이 나오는지 봐야 확정됨. 다음 점검(약 7/22) 때 최우선 확인 필요.

## 3. 배포 중 발생한 git 사고 및 처리

`git pull --rebase` 실행 중 작성팀 에이전트가 동시에 `blog/index.html`·`sitemap.xml`을
쓰고 있어 rebase가 unstaged 상태로 중단됨. **`git rebase --abort`로 안전하게 원상복구**
(내 커밋은 그대로 보존, 작성팀의 신규 글 파일 자체도 무손실 — untracked라 rebase 영향 밖).
이후 원격의 무관한 커밋(생필품 스냅샷, 겹치는 파일 없음 확인 후)과 `git merge`로 병합해
안전하게 재푸시함. 작성팀·홍보팀이 사라진 index.html/sitemap.xml 반영분을 정확히 인지하고
재적용하지 않은 채 마스터에게 이관한 판단은 적절했음 — 마스터가 최종적으로 새 글 카드/
사이트맵 엔트리를 직접 추가해 커밋.
